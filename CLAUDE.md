## 1. 项目身份

- **项目代号**:《假总统》(*El Doble*)
- **类型**:政治危机文字向叙事游戏，单局 3–5 小时，多周目设计
- **核心命题**:当一个体制需要任何人都可以替代，那么它真正在替代的，是什么？
- **基调**:拉美黑色幽默 + 魔幻现实主义的克制使用 + 良知与现实的反复磨损
- **不要做的**:英雄逆袭爽文、二元正邪剧、揭露真相后世界变好的天真叙事

## 2. 开发范式（核心约定）

### 技术栈
- **纯 HTML + JavaScript 单文件可玩 demo**。打开浏览器即跑，零安装、零构建。
- **不要引入 React / Vue / Svelte 等需要构建步骤的框架**。需要复杂状态时，先用原生 JS Class + 模块化 `<script type="module">`。轻量响应式可考虑 Alpine.js 或 Petite-Vue（CDN 引入），但**只在状态复杂到 vanilla JS 难维护时**引入。
- **样式**:原生 CSS，使用 CSS 变量做主题。**不要用 Tailwind 的 utility classes**——叙事游戏的视觉气质需要手写的细节。
- **存档**:`localStorage`，序列化为 JSON。**单存档 + 自动保存**（详见 `03_核心玩法.md`、`05_多周目机制.md`）。
- **构建/部署**:单文件优先；如需拆分，资源相对路径引用，确保 `file://` 也能跑。

### 开发方法论
- **架构原则:写第一段代码就为"扩展到全作"做准备**。即使现在只实现序幕，数据结构、事件系统、派系系统都要按全作骨架来设计。**反例**:把序幕的对话硬编码进 HTML——这是禁忌。
- **数据驱动**:事件、对话、派系、数值变动都是数据（JSON / JS 对象数组），引擎只负责消费。后期才方便交给我（Keyan）或外部协作者写内容。
- **垂直切片优先**:跑通序幕后，下一步先把 Acto I 第 1 天的所有循环（早班简报 / 强制项 / 突发事件 / 晚间简报 / 12 时辰）打通，**再做内容铺陈**。**当前阶段与里程碑状态见 `DEV_LOG.md`**。

### 代码与文本语言
- **代码（变量、函数、类、文件名）**:英文，遵循常规 JS 命名（camelCase / PascalCase）。
- **代码注释、commit message、PR 描述**:中文。
- **所有面向玩家的叙事文本**:中文为主，**允许并鼓励西语 flavor**（人名、地名、派系名、报刊名、咒骂、口头禅按设定文档保留西语原文，必要处给中文译注）。
- **不要把派系/角色名翻译成中文**。Doña Isabel 永远是 Doña Isabel，不要写成"伊莎贝尔夫人"。

## 3. 决策边界（AI 何时问、何时直接做）

**先问后做**（出现以下任意一项，必须征询）:
1. 引入新的依赖、库、框架。
2. 修改已实现的核心数据结构（事件 schema、状态机、派系系统）。
3. 改动玩法机制（数值轴、时辰消耗、Acto 划分等设定文档涉及的内容）。
4. 任何会影响"全作扩展性"的架构决定。
5. 不确定某段叙事文本的调性是否符合设定文档。

**直接做**（不问）:
- 命名、变量重构、文件拆分、目录结构。
- UI 微调、CSS、动效。
- 测试数据、占位文本（但要明确标注 `// TODO: placeholder` 中文化时）。
- Bug 修复（除非修复方式涉及上面"先问"的任何一条）。
- 写新事件 / 新对话的草稿——但**写完要让 Keyan review，不直接当成最终文本**。
- 在 `DEV_LOG.md` 记录刚做完的进度、风险、新决策——**不需要先问**。

## 5. 写作硬规则（来自 06_主题与调性）

> 写叙事文本前先在脑里检查这些。AI 出错最频繁的地方就在这里。

1. **三种声音不混用**:场景叙述者（第三人称、冷静）/ Saulito 内心独白（第二人称、斜体、口语）/ 新闻广播（拟文体）。同一段里只能有一种。其他替身的内心独白人称参 `角色档案/00_主角线/<对应替身>.md`（Catalina 第一人称广播稿、Tobías 第三人称传记体）。
2. **黑色幽默来自情境，不来自旁白**。**永远不要**写"哈哈，多么讽刺啊"这种贴脸笑的句子。
3. **魔幻现实主义是质感不是机制**。占卜婆婆的预言可以"准了一次"作为氛围，但**不能用来改写数值或解锁分支**。
4. **没有派系是"对的"**。每个派系都要有真实诉求和真实瑕疵。写派系视角时给出他们自己的"合理"。每个主要 NPC 至少要有一个让玩家真心想认同他的瞬间、一个真心讨厌的瞬间、一个意识到自己看错的瞬间。
5. **金钱要有体感**。提到数字时要有现实参照（参考 `01_世界观` 物价表）。"四万北元"应当让玩家感受到这是 Saulito 两年的收入，不是抽象数字。
6. **政治术语谨慎**。Saulito 是没受过政治训练的街头艺人，他的内心独白里不会出现"民粹主义""寻租"这种词——除非是他从别人嘴里听到的、半懂不懂的复述。
7. **不要解释隐喻**。设定里满是双关与暗喻（Veladuria 的字源、铜元、漫长之夜）—— **让玩家自己发现，不要旁白点破**。

**注意**:`角色档案/*.md`（设定层、给 AI 与 Keyan 读）与 `demo/content/npcs/*.json`（数据层、给引擎消费）**不是同一回事**。前者是写作辅助，后者是事件系统的数据。**不要混用，不要互相覆盖**。

**核心数据结构（早期就要立住的契约）**:
- `GameState`:当前 Acto / Day / Hour、数值轴（Cordura、Verosimilitud、Economía、Orden、Legitimidad、Tesoro）、派系态度表、隐性标记集合（`flags`）。
- `Event`:包含触发条件（哪天、哪时段、哪些 flag/数值前置）、文本（按三种声音分块）、选项（每个选项是 `effects: { state变动, flags新增, factionDelta, 后续事件入队 }` 的纯数据）。
- `DaySchedule`:早班简报组装、强制项 / 已申请觐见 / 死线 / 空白时段，由当日 events 与 flags 推导。

**禁止**:把文本写死在 `<div>` 里。**所有文本必须从数据层加载**，否则后期内容协作和翻译会全废。

## 7. AI 工作流默认行为

- **每次开新会话先扫一眼 `DEV_LOG.md`**——确定当前阶段、最近一次更新做了什么、有没有未解风险。再扫 `MEMORY.md`（如有）。**任何具体设定都以本仓库的设定文档为准**。
- **写代码前**:相关文档先 Read 一遍。不确定的地方先 Grep 找上下文。
- **写完代码**:在浏览器跑一次（用 `mcp__workspace__bash` 起个本地 server，或确认 `file://` 能跑），**至少手动走一遍最关键路径**。不能只 commit 不验证。
- **写完叙事文本**:拿 `06_主题与调性.md` 的"硬规则"和 `07_风格样本.md` 自检一遍——三种声音是否分清？是否有贴脸笑？金钱是否有体感？**写具体角色对白前**先翻 `角色档案/<对应分类>/<角色>.md` 的"声音卡"与"写作红线"。
- **commit 习惯**:小步提交，每个 commit 只做一件事。message 中文，开头标注模块（例:`engine: 时辰消耗系统支持负数检定`）。
- **大改动后追加 DEV_LOG**:做完一个里程碑节点 / 引入一个架构决定 / 解决一个长期风险——**追加一条到 `DEV_LOG.md`**（最近更新 + 必要时 ADR）。**这是默认行为，不需要每次问**。
- **不擅自更新设定文档**。如果 demo 实现暴露了设定矛盾，**列出矛盾并问 Keyan**，不要单方面改 `01_*.md` ~ `07_*.md`、`08_剧情/*` 或 `99_*.md`。
- **设定文档改动直接打印变更日志到聊天框**:每次改 `01_*.md` ~ `99_*.md`、`08_剧情/*` 或本 `CLAUDE.md`（任何**规则/设定**类 markdown）后，把"改了什么 / 为什么 / 影响哪些下游内容"以简明 changelog 的形式输出到对话里。**文档内部不保留 changelog / 版本历史 / 更新日期段落**——这些文档只给 Keyan 和 AI 看，可读性给 AI 的优先级高于人类读者，文件里只保留"当下生效的内容"，历史用 git log 追溯即可。
- **`DEV_LOG.md` 与 `角色档案/` 不发 changelog**——前者本身就是状态/历史的家，后者是从设定派生的工作汇编。但当原始设定文档有改动时，要回头同步对应角色卡（参 `角色档案/_索引.md` §五·维护纪律）。
